CORE ONE

Thinking like a developer

This lesson you will learn about what it is like to think like a developer. Thinking in this mindset takes problem solving skills and persistence to achieve good results. We will look into this mindset using the metaphore of ultramarathon running.

Describe the process for solving problems?

The following process for thinking like a web developer reflects that running an 100 km Ultramarathon. I ran my first ultramarathon this year.
Why? When running a 100km marathon, it can look very daunting even impossible to the human mind.
The first step to completing a ultramarathon is becoming intentional. Telling your mind that this is what my goal is, and this is how i will achieve it. Once you have committed to this idea of completing your goal in your mind, the correct actions will follow.
The first action is plan ahead, figure out where you want to have breaks, how much food you will bring on the marathon, when you will eat to keep your nutrition and energy up? Once you have a plan on how to tackle the marathon it looks alot more achievable.
Then you put the plan into action and break the ultramarathon down into mini stages, focusing on 1km at a time. This helps focus your mind and energy on the small steps, instead of becoming overwhelmed by the bigger picture.

Over the course of the Ultramarathon, you will become so tired you wont want to continue. In that moment you realise your brain wants to give up, so to overcome this mental breakdown you look at other alternatives that will help you dig deep and keep going. Like Web development, if you feel stuck you look at alternatives and keep going until you solve it.
In the last 10km of the race, you have used all the tricks in the book, and you are almost there. Then you figure one last trick to keep pushing forward, this little trick is seeing your family 5km from the finish line. This motivates you to keep going like nothing else. Once you complete the ultramarathon all you can do is think about it.You think about the entire marathon and reflect on all the highs and lows, and what you do different next time. Similar to development, when you finish solving a problem, you reflect on all the pitstops and speed-boosts that you got you to the finish and how you could improve in the next problem to come.

Becoming clear on the problem is one of the most important things to do when tackling a problem. Not only will this give you a clear idea of whats ahead, this will help you see potential pitfalls and possible solutions before you start.

TECH TWO

Version Control with Git - Primer

This lesson you will explore a what Github is and the difference between Git and Github.

Whats the difference between git and GitHub?

Firstly we need to understand what version control is. Version control allows you to work on files in folders, without your changes overriding other peoples work. The version control will tell you if your going to override someones work.
Git is a system for version control, it runs with the command line on your local machine, and it allows you to keep track on your files and folders and any changes that have happened in the repository.

this is a secret header, no one can see

Git essentially does this by creating “timestamps” on the files you are working on for each person who is working on it. This makes it easy to reverse in time and see what you were working on, or what one of your time mates were working on.
Github is a website where you can upload your git repository online. You should do this to backup your files, It gives you a visual interface to navigate to your folders. It gives other people a way to navigate your repos, it makes repo collaboration easy.

What is an analogy for Github and Git?

Traditionally, to prevent users from overwriting each-other work they would create many versions and share them. This can get very confusing later down the track when there are many files in folders. Instead of creating many versions, Git creates “timestamps “ each time someone in the team has made a change.

this is a secret header, no one can see

A person can then click on the time stamp and reverse or fast forward the project to see that persons changes. Github does the exact same thing, but on steroids. Github is an online website that has a graphical interface so its easy to navigate, and it has a very large amount of storage for you to share and upload your Git repositories.

What is a GitHub work flow?

Git hub flow is creating a branch when working on a feature or an issue, then making changes and making a pull request to review you changes, then pushing your changes to be approved to merge the branch with the master.

How github enables developers to collaborate?

1. Create a directory and use git init to make it a git repo.
2. Then check the status for a quick update.
3. Add files into the repo
4. Track these files with git by placing them into the staging index with git add
5. Commit these files files into the working directory, by entering git commit.

It will then open your code editor where you can do your first commit which is usually “initial commit” now your files are being tracked by git. For every change you do to each file, you should be committing them to git.

TECH FIVE

Branch, Pull, Merge Primer

This lesson you will learn how to use Git to branch, push, and merge.

What is a Master?

Whenever you create a new repository, it will become the master. Later on you will start branching off the master to fix issues, create new features and so on. The master is expected to have the most perfect code, this is the repository that is closest to the finish product.

Why create a Branch?

We create branches when we want to fix issues, or create new features without effecting the master. Once we have fixed the issue, we can the ask the owner of the master to review the fix and approve merging it with the master. This stream of collaboration is also known as the github flow. It allow changes to be made in a fast, but safe enviroment.

TECH SEVEN

GitHub Fork & Clone

This lesson you will learn about forking and cloning to your local machine from Github.

What is the process to fork and clone?

1. First you must fork the original repository
2. Go into settings and make sure the issues box is checked
3. Clone the repository, do this by copying the link and going to iterminal and pressing "git clone (Url of repository)"
4. make sure you have prepared a directory for this clone, and are in the correct directory in iterminal when making the clone.
5. Go in and make any changes to the files that you like.
6. Go back to iterminal
7. Check the git status
8. Add the changes to the stageing index by going git add (filename)
9. Commit the changes by going git commit -m
10. Push the changes by going git push origin master
11. Your changes should be in.

What are some examples of when you would fork?

Forking follows the same structure as branching, it is good to fork when you want to make a new feature, or fix an issue without effecting the main repository. However, forking enables you to make a copy of your repository so when you push changes. you do not need to wait for an owner to approve your changes, as you are the owner. The owner of the main repository can then view your repository and decide whether or not they would like to merge. This workflow enables large organic teams to work effecively and safely without making overrides. This stream of collaboration is also known as the github flow. It allow changes to be made in a fast, but safe enviroment.

TECH EIGHT

Git Set Remotes and Merge Prime

This lesson you will learn about remote and local copies of code and how to use them effectively. In this lesson we will also cover more about merging.

What is the difference between cloning from a forked and cloning from a non-forked repo?

A fork replicates the existing repository. Forking a repository allows you to freely experiment with changes without affecting the original project. When you clone from a forked, all changes that you decide to push later on will effect your copy. This means that if you make any mistakes . . . its ok because your using a copy of repo.

this is a secret header,

no one can see

If you have cloned a non-forked repo, every change you commit and push will effect the original copy. This makes it very risky for if you make mistakes to the original copy there is no going back.

What is a analogy to describe forking?

Forking is like branching, instead of editing the original... you can branch off and edit in a safe enviroment, then submit a pull request ready for the owner to look at. if they like it they might push and approve your changes then merge it into the original.

What is a remote?

Remote repositories are only thought as a means for sharing and exchanging code between developers - not for actually working on files. Local repositories reside on the computers of team members. In contrast, remote repositories are hosted on a server that is accessible for all team members - most likely on the internet or on a local network. A remote repository doesn't have such a working directory: it only consists of the bare ".git" repository folder.
